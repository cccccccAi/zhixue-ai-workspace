# 智学网 PRD 生成器

## 执行协议

- 产品同行语气，PM 是产品总监级别，不教基础理论
- **全部中文输出**，不用英文 User Story 格式
- 每次输出 ≤ 15 行（不含代码块和表格），超过时用 AskUserQuestion 过渡
- 产出文件命名：`prd-{功能缩写}.md`，存入 `./learning/outputs/`
- md 文件生成后用 `open -a "MarkText"` 自动打开
- 不确定的内容**必须用占位符**，严禁用模拟数据伪装成真实数据

---

## 启动流程

### 1. 解析输入

解析 `$ARGUMENTS`：

- **有内容** → 作为功能名称/问题描述，直接进入 Step 1
- **空** → 用 AskUserQuestion 问"你想做什么功能？"

### 2. 检测训练产出（增强包）

自动检测 `./learning/outputs/` 目录下的训练产出文件：

```
ls ./learning/outputs/level-1*.md
ls ./learning/outputs/level-2*.md
ls ./learning/outputs/level-3*.md
ls ./learning/outputs/level-4*.md
ls ./learning/outputs/level-6-zhixue-data-analyst/
```

- **存在** → 读取并提取与当前功能相关的素材（见素材提取映射）
- **不存在** → 使用内置领域知识（[zhixue-domain.md](../../domain/zhixue-domain.md)），提示："将使用内置的智学网领域知识。如果你做过 `/learn-pm-data` 闯关训练，产出会自动丰富 PRD。"

### 素材提取映射

| 训练产出            | 提取内容                     | 填入 PRD 章节                            |
| ------------------- | ---------------------------- | ---------------------------------------- |
| level-1（指标体系） | North Star、L1/L2 指标、OKR  | 第 2 节目标、第 10 节成功指标            |
| level-2（用研综合） | 核心发现、用户画像、用研引言 | 第 1 节问题定义、第 4 节用户场景、附录 A |
| level-3（竞品分析） | 竞品优劣势、战略建议         | 第 1 节竞争压力、第 3 节不做什么         |
| level-4（数据叙事） | 数据洞察、受众适配           | 第 1 节证据来源                          |
| level-6（数据技能） | 实体定义、指标精确口径       | 第 8 节数据依赖、第 10 节统计口径        |

提取时只保留与当前功能名称/主题相关的段落，不全量搬入。

## 五步工作流

### Step 1：需求锚定（1 分钟）

用 3 个 AskUserQuestion 快速定框。每个单独问，不要一次问 3 个。

**第一个**：驱动力

```
这个需求的驱动力是？
- 指标异常（某个数据在恶化）
- 用研发现（访谈/问卷发现的痛点）
- 竞品压力（竞品做了我们没做的）
- 战略方向（领导层决策/OKR 要求）
- 体验优化（日常改进）
```

**第二个**：时间约束

```
目标上线时间？
- 本季度（< 3 个月，需求要精简）
- 下季度（3-6 个月，正常范围）
- 半年以上（可以做大一些）
- 还没定，先写 PRD 再评估
```

**第三个**：读者

```
这份 PRD 主要给谁看？
- 设计师（需要详细场景和交互说明）
- 研发工程师（需要数据依赖和接口需求）
- 领导/决策层（需要问题定义和 ROI 论证）
- 设计+研发都要看（完整版）（推荐）
```

根据回答调整 PRD 侧重：

- 读者含设计师 → 第 5 节（页面流程）必须详细
- 读者含研发 → 第 8 节（数据依赖）必须详细
- 读者是领导 → 第 1 节（问题定义）和第 10 节（成功指标）加强
- 时间紧 → P0 控制在 4 个以内

---

### Step 2：素材提取（自动）

如果检测到训练产出，自动读取并提取。提取完成后输出简要说明：

```
检测到 X 份训练产出，已提取相关素材：
- 指标体系：North Star = [xxx]，L1 含 [xxx]
- 用研发现：[核心发现摘要]
- ...

这些素材将自动填入 PRD 对应章节。
```

如果未检测到训练产出：

```
未检测到训练产出，将使用内置领域知识生成 PRD。
```

---

### Step 3：PRD 骨架生成（5 分钟）

基于 [prd-template.md](references/prd-template.md) 生成 12 章节 PRD。

**章节列表**：

1. 问题定义（含根因分析表、证据来源）
2. 目标（SMART 格式，3-5 个）
3. 不做什么（3-5 个，每个有原因）
4. 用户场景与验收标准（按角色拆分，"当...应..."格式）
5. 页面流程与信息架构（ASCII 线框图 + 流程图，如果读者含设计师）
6. 业务细分（学科/角色差异化内容，如果功能涉及）
7. 需求详细规格（P0 ≤ 6 个 + P1 + P2）
8. 数据依赖与接口需求（数据 | 状态 | 动作 | 负责人 | 截止）
9. 边界情况与降级方案（≥ 3 个场景）
10. 成功指标（先行 + 滞后 + 成功/失败判断）
11. 待决事项与推进计划（每项有建议方案 + 负责人 + 截止）
12. 里程碑计划（关键路径标注）

**生成规则**：

- 有训练素材的章节 → 预填训练内容，标注 `[来源: 闯关训练第X关]`
- 需要真实数据的地方 → 用 `[待填: 描述]` 占位
- 需要确认的假设 → 用 `[待确认: 描述]` 占位
- 第 5、6 节根据功能类型判断是否需要（不涉及页面/学科差异则省略）
- 角色术语严格参考 [zhixue-domain.md](../../domain/zhixue-domain.md)

**生成后**：

1. 用 Write 工具写入 `./learning/outputs/prd-{功能缩写}.md`
2. 用 `open -a "MarkText"` 打开文件
3. 输出 PRD 概览（章节列表 + 占位符数量）

---

### Step 4：真实数据填充（10-15 分钟）

按 [data-filling-guide.md](references/data-filling-guide.md) 的交互脚本执行。

**核心流程**：

1. 扫描 PRD 中所有占位符，输出"数据缺口清单"
2. 用 AskUserQuestion 询问 PM 是逐项填充还是跳过
3. 如果填充：按类型分批（每批 3-4 项），三种应答路径：
   - "我知道" → 替换占位符
   - "需要查" → 保留并标注"评审前补充"
   - "改为估算" → AI 辅助推算，标注为估算值
4. 汇总所有 `[待确认]` 项 → 生成"附录 B：跨团队确认清单"追加到 PRD 末尾
5. 用 Edit 工具更新 PRD 文件

---

### Step 5：交付质检（2 分钟）

按 [delivery-checklist.md](references/delivery-checklist.md) 的 20 项清单自动检查。

**检查方式**：

- 读取 PRD 文件内容
- 逐项检查（搜索占位符、检查章节结构、验证格式）
- 输出质检报告（✅ 通过 / ⚠️ 待处理 / ❌ 阻塞）

**阻塞项处理**：

- 如有 ❌ 阻塞项 → 用 AskUserQuestion 逐个引导修复
- 修复后重新检查该项
- 全部阻塞项清除后输出最终结论

**最终输出**：

1. 用 Edit 工具写入最终版 PRD
2. 用 MarkText 重新打开
3. 输出完成提示：

```
PRD 已就绪。建议下一步：
1. 发送跨团队确认清单给对应负责人
2. 预约设计/研发评审会
3. 如需修改，在 MarkText 中直接编辑或再次运行 /write-prd
```

---

## Anti-Patterns

- **不要用模拟数据伪装成真实数据**：宁可留占位符，不可编造数字
- **不要教 PM 基础理论**：不解释什么是 P0/P1/P2、什么是用户故事
- **不要一次输出整份 PRD**：PRD 很长，骨架生成后用 MarkText 打开让 PM 浏览，AI 输出概览即可
- **不要跳过数据填充直接质检**：数据填充是核心差异化，至少展示缺口清单
- **不要修改训练产出文件**：只读取训练产出，PRD 是独立新文件
- **不要在填充环节反复追问**：PM 说跳过就跳过
- **不要用英文写用户故事**：全部中文，"作为XX，我想YY，这样我能ZZ"
